# Auditoría UI — Ocean Breeze Fase 0

**Fecha**: 2026-07-14
**Autor**: agente (orquestador) en PLAN.md Fase 0
**Estado**: pre-Fase 1 (input para decidir top 3-5 pantallas prioritarias)

## Objetivo

Identificar qué falla en las pantallas actuales vs `DESIGN.md` antes de empezar cualquier Fase 1 de rediseño. Produce una lista priorizada por severidad para guiar la decisión de top 3-5 pantallas que más duelen (una de las 4 preguntas abiertas de `PLAN.md`).

## Metodología

1. Lectura completa de `DESIGN.md` (989 líneas).
2. Mapeo de `src/app/**/page.tsx` → lista de 36 pantallas del `PLAN.md` (§Inventario).
3. Greps sistemáticos por anti-patrones conocidos:
   - `shadow-*` fuera de UI primitives / landing exception.
   - `bg-{emerald,red,amber,blue,green,rose,indigo,orange,yellow,sky,cyan,violet,fuchsia,pink}` (paletas Tailwind v3 en vez de tokens semánticos).
   - `MetricCard | StitchKpiCard | ExecutiveKpiCard` (drift del primitive KPI — debe ser `KpiCard`).
   - `<tr ... cursor-pointer>` y `onClick={...navigate}` en filas (anti-pattern §9).
   - `<Card>` envolviendo `<DataTable>` (anti-pattern §Card wrapping).
   - `<DialogContent>` sin `w-[95vw]` (regla §2 modales).
   - Hex hardcoded `#xxxxxx` (regla §4 + §5 tokens semánticos).
   - `bg-slate-900|800` / `bg-zinc-*` (chrome no migrado).
4. Spot-check visual de páginas canónicas: `/dashboard`, `/admin`, `/reports`, `/settings`.
5. Conteo de uso de primitives canónicos: `<KpiCard>`, `<DataTable>`.

## Hallazgos priorizados

### 🔴 S — Severidad alta (afecta UX o consistencia)

**S1. [FUNCIONAL] Pricing (#2) no existe como ruta**
- **Impacto**: Owner no tiene forma visible de comparar FREE vs PRO. Solo ve tooltips en `/reports`. Fricción para upgrade.
- **Recomendación**: Crear `/pricing` (página pública) con tabla comparativa + CTA a `/register`. Rediseño visual ya cubierto por el sistema Ocean Breeze.

**S2. [FUNCIONAL] Forgot password (#5) no existe**
- **Impacto**: Si un owner olvida su password, no hay flujo de recuperación. Pierde acceso a la cuenta.
- **Recomendación**: Implementar flujo canónico Next.js (form email → mailto con reset token → nueva password). Probablemente requiere Resend (ya configurado para PRD-0003).

### 🟡 M — Severidad media (drift menor o desviación arquitectónica)

**M1. [UI] Detalle de pago standalone (#19) no existe**
- **Impacto**: El owner ve pagos en 2 lugares (tabla en `/payments` + dentro de `reservation-detail-dialog`). PRD-0002 agregó Copy/Regenerar/Delete pero no consolidó un "Detalle de pago" con comprobante + acciones + history.
- **Recomendación**: Crear `payment-detail-dialog.tsx` reutilizable (estilo `reservation-detail-dialog`), invocable desde `/payments` y desde `reservation-detail-dialog`. Patrón full-bleed `w-[95vw] max-w-2xl gap-0 p-0 overflow-hidden`.

**M2. [ARQUITECTÓNICO] iCal setup vive en detalle de propiedad (#22)**
- **Impacto**: Setup de Calendarios Externos está dentro de `/properties/[id]`. Funcionalmente OK pero no matchea el PLAN.md y complica el journey para owners con varias propiedades (tienen que entrar a cada una).
- **Recomendación**: Decisión producto. Opción A: dejarlo como está (per-property, simple). Opción B: crear `/integrations/calendars` agregado con lista de todas las propiedades. Recomiendo A si owner promedio tiene <5 propiedades.

**M3. [FUNCIONAL] Sync log no visible para owner (#23)**
- **Impacto**: Si un calendar externo falla al sincronizar, el owner no lo sabe. Solo lo descubre cuando una reserva manual colisiona con un bloqueo externo.
- **Recomendación**: Agregar panel "Estado de sincronización" dentro del detalle de propiedad, mostrando `lastSyncedAt` + `lastSyncError` por calendar. Bajo esfuerzo, alto valor.

**M4. [UI] Recordatorios de pago (#21) no son una vista dedicada**
- **Impacto**: PLAN.md lista "Recordatorios de pago" como pantalla #21. Implementado vía notificaciones in-app (PRD-0003) pero no como vista consolidada.
- **Recomendación**: Decisión producto. Si el inbox in-app cubre el caso, dejar Out of Scope (moverlo al backlog v2 como "Inbox page `/notifications`"). Si se quiere vista consolidada, crear `/payments/reminders`.

**M5. [UI] 3 `<DialogContent>` sin `w-[95vw]` en admin**
- **Ubicación**: `src/components/admin/admin-users-client.tsx:610, 720, 755`
- **Impacto**: En mobile, el dialog ocupa el ancho completo del `DialogContent` default, no respeta el patrón §2.
- **Recomendación**: Agregar `className="w-[95vw] max-w-md"` a los 3 (o `max-w-lg` según contenido). Verificar tamaño real de cada uno.

**M6. [UI] `bg-violet-500/5` decorativo en admin/users/[id]**
- **Ubicación**: `src/app/admin/users/[id]/page.tsx:182`
- **Impacto**: Es un blur decorativo (`-left-12 -bottom-12 blur-3xl`). Es la única instancia de `bg-violet-*` en producción. Rompe la regla §5 (todo color hex debe pasar por tokens semánticos).
- **Recomendación**: Cambiar a `bg-primary/5` o `bg-info/5` para armonizar con el resto.

### 🟢 L — Severidad baja (nit, no urgente)

**L1. Naming inconsistency en `/admin/users/[id]` (tabs)**
- "Propiedades" / "Reservas" / "Pagos" / "Tickets" — usar `<Tabs>` con misma jerarquía visual que `/support/[id]` para consistencia.

**L2. Falta de breadcrumbs en rutas profundas**
- `/admin/users/[id]`, `/properties/[id]`, `/support/[id]` no muestran breadcrumb "Volver a [sección]". Fricción menor de navegación.

**L3. Páginas sin "Última actualización" timestamp**
- `/reports` y `/payments` muestran datos que pueden quedar stale entre refreshes. Agregar `lastUpdatedAt` discreto al lado del PageHeader.

Basado en esta auditoría, las 3-5 pantallas que más impacto darían en Fase 1:

| Prioridad | Pantalla | Razón |
|---|---|---|
| 1 (alta) | `/pricing` (S1) | Funcional, marketing crítico para conversión FREE→PRO. Costo: 1 sesión. |

| 5 (baja) | Drift cleanup (M5/M6 + L1/L2) | Cosmético, batcheable. |

**Mi apuesta al top 3**: pricing + forgot-password + sync log de iCal. Las 3 son funcionales (no cosméticas), costo bajo por sesión, y desbloquean journeys críticos (conversión, recuperación de cuenta, debugging de integraciones).

Si el usuario quiere **refrescar visualmente** pantallas existentes (Fase 1 "puramente estética"), mi apuesta sería: ninguna. El codebase ya está al 90% de Ocean Breeze y los 3 grandes layouts (`/dashboard`, `/reports`, `/admin`) ya están canónicos. Refrescar más sería busywork.
